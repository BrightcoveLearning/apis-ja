    概要:CMS API

    このトピックでは、CMS API の概要について説明します。 CMS API は、データへのキャッシュされていない読み取りアクセスを提供します。 これは、CMS API を使用してビデオや再生リストを変更し、データをすばやく読み込んで正しいことを確認するため、時間を重視するパブリッシングワークフローでは重要です。

    API リファレンス

    「 API リファレンス 」も参照してください。

    一般情報

    ベース URL

    CMS API のベース URL は次のとおりです。

     https://cms.api.brightcove.com/v1

    アカウントパス

    いずれの場合も、特定の Video Cloud アカウントに対してリクエストが行われます。 したがって、あなたは常にベースURLにアカウントIDが続く用語アカウントを追加します：

     https://cms.api.brightcove.com/v1  /accounts/ {account_id}

    認証

    要求の認証には、 認証ヘッダー が必要です。

     承認:ベアラー {access_token}

    access_tok en は一時的な OAuth 2 アクセストークンであり、 ブライトコーブ OAuth サービスから取得する必要があります。 詳細については、「 ブライトコーブ OAuth の概要 」を参照してください。

    クライアント認証情報を作成する最も簡単な方法は、Brightcove Studio の管理ページを使用することです。 詳細な手順については、「 API 認証資格情報の管理 」を参照してください。

    オペレーション

    クライアント認証情報を要求するときは、必要なアカウントアクセスまたは操作の種類を指定する必要があります。 CMS API で現在サポートされている操作の一覧を次に示します。

    • 動画データ:

      ビデオクラウド/ビデオ/
      ビデオ読み取り       /ビデオクラウド/ビデオ/
      ビデオの作成       /ビデオ/
      ビデオ/更新       ビデオクラウド/ビデオ/削除
      または
      ビデオクラウド/ビデオ/すべての
       ビデオクラウド/ビデオ/アセット/削除 （のみが必要です。Studioで認証情報を作成するときに、この権限を取得 できない ことに注意してください。 OAuth API または Brightcove ラーニングサービスによって作成されたサンプルアプリ を通じて行う必要があります）。

    • プレイリストのデータ:

      ビデオクラウド/プレイリスト/
      ビデオクラウド/       プレイリスト/ビデオクラウド/ プレイリスト/
      ビデオクラウド/プレイリスト/更新
       ビデオクラウド/プレイリスト/削除
      または
      ビデオクラウド/プレイリスト/すべて

    • 通知:
      • ビデオクラウド/通知/すべて （ 通知 用）

    レート制限

    この CMS API は、データへのキャッシュされていない読み取りアクセスを提供します。 これは、 CMS API を使用してビデオや再生リストを変更し、データをすばやく読み込んで正しいことを確認するため、時間を重視するパブリッシングワークフローでは重要です。

    CMS API は、高スケールのランタイム使用には適していません（トラフィックの多いパブリック Web ページ上の動画へのアクセスなど）。 トラフィック量の多いアプリケーションでは、 Playback API 、ギャラリー、プレーヤー、ネイティブ SDK などのキャッシュされたインターフェイスを使用する必要があります。

    Video Cloud システムのパフォーマンスを確実にするために、 CMS API への同時呼び出しはアカウントごとに20回まで許可されています。 アクセス頻度は、1 秒あたり 10 リクエスト未満にする必要があります。

    同時呼び出しは、並列で実行されている複数のプロセスまたはアプリケーションが API リクエストを作成している場合に発生します。 リクエストの完了に 1 秒以上かかることがあるため、1 秒あたり 10 件を超えるリクエストを作成していない場合でも、同時実行制限を超える可能性があります。

    複数のアプリケーションが 1 つのアカウントの CMS API に統合される場合、これらのアプリケーションには、同時実行制限またはレート制限に達したインスタンスを処理するためのバックオフおよびリトライロジックが必要です。

    レートまたは同時実行制限のいずれかが超過した場合、 429-TOO_MANY_REQUESTS エラーが返されます。

    参照 ID の競合

    参照 ID の一意性を保証するために、CMS API は、割り当てられているビデオの操作後、最大 3 分間 ID をロックします。 これにより、失敗するリクエストを再試行しようとした場合、または以前に割り当てられていたビデオを削除してからすぐに参照 ID を再利用しようとすると、409 エラーが返されることがあります。 詳細については、「 エラー・メッセージ・リファレンス 」を参照してください。

    動画アセットの制限

    動画あたりのアセット数は 1,000 件に制限されています。 アセットには、レンディション、オーディオトラック、テキストトラック、画像のほか、デジタルマスターが含まれます。 レンディションと画像の合計アセットが 10 ～ 15 個を超えることはめったにありません。そのため、150 種類の言語に対応したオーディオトラックとテキストトラックを別々に持っていても、アセット数は 350 個以下になります。

    使用上の注意

    CMS API は、統合および公開ワークフローを対象としています。 パブリック Web ページからこの API へのクライアント側の呼び出しを行うことはできません。そうすると、API のクライアント認証情報が公開されます (このため、API は CORS に対応していません)。 API 呼び出しをサーバー側のプロキシ経由でルーティングする場合は、クライアント側 (Web) アプリから呼び出しを行うことができます。 Brightcove Learning Services では、このアプローチを使用するいくつかの サンプルアプリ と、 この種のアプリを構築するためのガイド を提供しています。

    すべてのビデオまたは大規模なビデオセットを取得する場合は、 大きなデータセットに関するメモ を参照してください。

    ハードコードされた URL を避ける

    サムネイル、ポスター、動画ファイル、その他のメディアの URL は、ページやアプリケーションでハードコーディングしないでください。 CMS API は常にメディアファイルの現在の URL を返しますが、URL 自体は変更される可能性があります。 CMS API （または Playback API ）リクエストを使用して、ページが読み込まれるたびにこれらの URL を取得するか、6 時間以内にキャッシュしてください。

    ビデオと画像の URL のキャッシュ

    動画や画像の URL をキャッシュしてページのパフォーマンスを向上させることはできますが、キャッシュは定期的に更新する必要があります。 ページのパフォーマンスを向上させるために取得した URL をキャッシュする場合は、少なくとも 6 時間に 1 回は API 呼び出しを繰り返してキャッシュを更新してください。

    GET リクエストパラメータ
    パラメータ 説明
    q 検索用のクエリ文字列-詳細については、「 検索構文 」を参照してください。
    限界 返される動画の数-1 ～ 100 の整数である必要があります。 デフォルト:20
    オフセット スキップする動画の数（ページング結果用）。 正の整数である必要があります。 デフォルト:0
    ソート 並べ替えの基準となるフィールドを指定するストリング。 降順でソートするには、- で始まります。 q の値を指定した場合、デフォルトのソートは「score」（検索結果と元のクエリの関連性）によるものです。 q の値が指定されていない場合、デフォルトのソートは updated_at 降順です。 ソートには、 名前 、 参照ID 、 作成済み 、 公開 中、 更新 中、 スケジュール開始時 、 スケジュール終了時 、 状態 、 再生 中、 再生中week

    動画を検索する

    ブライトコーブの CMS API は、動画 クラウド ライブラリで動画を検索するためのプログラム的な方法を提供します。

    ビデオデータの基本的な検索や複雑な検索を実行するには、 q パラメータを使用します。

     https://cms.api.brightcove.com/v1/accounts/921483702001/videos?  q= {検索用語}

    ビデオを検索する方法の詳細については、「 ビデオの検索 」ドキュメントを参照してください。

    プレイリストでは、検索文字列でサポートされている値がより限定されます。 現在、 タイプ 、 名前 、 説明 、および re ference_id で検索できます。 有効な検索の例をいくつか示します。

    • Q=タイプ:明示的
    • Q=タイプ:activated_oldest_to_newest
    • Q=タイプ：アルファベット順
    • q=クマ+カワウソ （名前、説明、または参照IDには「クマ」または「カワウソ」のいずれかが含まれている必要があります）
    • q=%2bname: ベア+タイプ:明示的 (名前には「ベア」が含まれている必要があります)

    詳細については、「 プレイリストの検索 」を参照してください。

    再生可能：true/false という検索用語に注意してください。再生可能（または再生できない）動画のみを返すために使用できます。
     https://cms.api.brightcove.com/v1/accounts/:account_id/videos？ q=%2B再生可能:真

    このクエリでは、返されたビデオが再生可能であること (非アクティブ化されていないこと、スケジュールされていないこと、再生可能なレンディションがないこと) が必要です。

    再生できない動画を検索する場合は、 q=%2BPlayable: false を使用します。

    ページングの結果

    limit パラメーターを使用して、リクエストで返されるアイテムの数 (最大 100) を指定します。 次に、 offset パラメーターを使用して、 制限 を超える結果セットをページングできます。 オフセット は、スキップする項目の数です。

    たとえば、次の検索では、合計結果セットの動画が 75 件以上あると仮定して、合計結果セットの動画が 51～75 件返されます。

     /動画？ q=updated_at:2014-01-01.. 2014-06-30&limit=25&offset=50

    制限 パラメータと オフセット パラメータは、ビデオとプレイリストの両方に使用できます。

    注：再生リストに属する動画をリクエストする場合、 制限 と オフセット は機能 しません 。このリクエストは常に再生リスト内のすべての動画を返します。

    ページングのベストプラクティス

    CMS API で同時に変更操作が行われる可能性があるため、結果セットをページングするときは、次の手順を実行することをお勧めします。

    1. カウント リクエストを実行して、結果セット内の動画の総数を取得します。
       /アカウント/578380111111/カウント/動画？ q=タグ:自然
    2. 結果セットからデータのグループを返すには、 limit パラメータと offset パラメータを使用します。
       /アカウント/578380111111/動画？ q=タグ:nature&limit=20&offset=50
    3. ページによっては、動画が 20 個未満になる場合があります。 最初の カウント リクエストと同じ数の動画を要求すると、結果セットの最後に到達したことがわかります。

    要約すると、ビデオ数が元の カウント 要求と同じになるまで、ページを取得し続けます。この数は過大評価の側で誤っているからです。 質問:

     カウント/ページサイズ + 1 ページ

    ビデオ結果の並べ替え

    パラメータ sort=field_name を使用して、結果のソート方法を指定します。ビデオとプレイリストの両方をソートできます。 次のビデオフィールドで並べ替えることができます： [1-1]

    降順の場合は、フィールド名の前にマイナス記号 ( sort= - field_name) を付けます 。
    • 名前
    • reference
    • created
    • published
    • updated_at
    • schedule_starts_at （注：これは ソート フィールドです。 検索フィールドは schedule.starts_ atです ）
    • schedule_ends_at （注：これは ソート フィールドです。 検索フィールドは schedule.ends_ atです ）
    • 状態
    • plays_total [1-2]
    • plays_trailing_week [1-2]

    注意事項

    • [ 1-1] 動画検索コールにソート値を指定しない場合、結果は関連性でソートされます。 GET ビデオ呼び出しにソート値を指定しない場合、結果は updated_at 降順で ソートされます。
    • [ 1-2] あなたは plays_total または plays_trailing_week でソートすることができますが、これらのフィールドは結果に含まれていません

    プレイリストの結果を並べ替える

    プレイリストは、次のフィールドでソートできます。

    • 名前
    • updated_at (デフォルト)
    降順の場合は、フィールド名の前にマイナス記号 ( sort= - field_name) を付けます 。

    すべてのビデオと大規模なデータセット

    アカウント内のすべての動画を取得する場合、または多数の動画を取得する場合は、次の点に注意する必要があります。

    1. 最大許容 制限 （100）を使用するよう誘惑されるかもしれませんが、API リクエストがタイムアウトする可能性を最小限に抑えるために、25 以下のバッチで動画を取得することをお勧めします。
    2. 大きなデータセットをページングすると、操作中にビデオデータが更新され、応答の項目がずれる可能性があります。
      • 連続するページにアイテムが繰り返し表示される場合があります。
      • 前の応答セットに移動したため、アイテムが失われる可能性があります。

      最初の可能性を考慮するには、動画の取得が完了した後、アプリは項目リスト全体を重複除外する必要があります。 2番目の可能性を処理するには、取得されたアイテムの総数（重複除外後）を予想していた数と比較してから、リクエストを再実行し、結果をlast_modified_date（降順）でソートする必要があります。欠落したアイテムを取得するために複数のバッチを取得する必要はありません。

    3. 返された結果を適切にソートすることで、前の項目のシナリオの可能性を減らすことができます。 検索 の関連性 によるデフォルトのソートは、キーワード、タグ、およびカスタムフィールド値の組み合わせを検索する複雑なアルゴリズムに基づいています。 複数のキーワード、タグ、カスタムフィールドに基づいて動画を検索する場合は、関連性に基づいて並べ替えることができます。 ただし、動画のすべてまたは大きなセットを取得しようとするだけの場合は、 sort パラメータを明示的に設定することで、返される項目の順序をより細かく制御できます。

    資産

    アセット操作を使用すると、レンディション、マニフェスト、画像、テキストトラックなどのアセットを管理できます。 アセットを取り込むには、 動的取り込み API を使用する必要があります。 CMS API /アセット の POST および PATC H オペレーションを使用して、 リモートアセット を追加および更新できます。 CMS API GET オペレーションは、取り込まれたアセットとリモートアセット の両方 で機能します。

    • レンディションの追加、更新、削除
    • デジタルマスターのメタデータを追加、更新、または削除する
    • HLS などのセグメント化されたビデオタイプのマニフェストを追加、更新、または削除する
    • ポスターおよびサムネイル画像を追加、更新、または削除する
    • WebVTT テキストトラックまたは DFXP キャプションを追加、更新、または削除する

    通知

    ビデオライブラリで video_change イベントが発生したときに通知を受け取ることができます。 指定したURLに通知が送信されます。このURLは、 HTTP POST リクエストを処理できるアプリケーションを指している必要があります。

    注意事項:
    • ビデオのメタデータを変更すると、 video_change イベントがトリガーされます。 動画で使用されているアセットを変更しても、変更イベントは発生 しません 。
    • 通知に加えて、ビデオに関連付けられたインジェスト（取り込み、置換、または再トランスコード）ジョブのステータスを取得できるステータス API があります。

    通知の失敗

    通知システムは、カスタマーサーバからの 4xx または 5xx のリターンを再取得可能なエラーとして扱います。 失敗したコールバックは最大20回再試行され、後続のコールバック間の遅延は指数関数的に増加します。 最初の数回の再試行は、最初のコールバック試行から数分以内に行われます。 コールバックが失敗し続け、20回目の再試行まですべての方法を取得した場合、再試行遅延は数日間になります。

    video_change イベントは、自動化されたシステムプロセスとユーザーアクションによってトリガーされるため、これらのイベントは、StudioまたはAPIを介してビデオプロパティに加えた変更に必ずしも対応するとは限りません。

    ファイアウォール

    ファイアウォールを通過する着信トラフィックのソースに関する厳格なポリシーが組織にある場合、すべての AWS East リージョン IP が許可されます。 これは変更される可能性があるため、すべての AWS IP をホワイトリストに登録する必要があります。 詳細は https://docs.aws.amazon.com/general/latest/gr/aws-ip-ranges.html を参照。

    通知操作

    現在通知に使用できる操作は次のとおりです。

    • サブスクリプションの作成
    • 1 つまたはすべてのサブスクリプションを取得する
    • サブスクリプションの削除

    通知操作の詳細については、 API リファレンス を参照してください。
